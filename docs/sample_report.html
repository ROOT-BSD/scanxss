URLhttps://demo.site.com/run Параметрcmd Payload; id Доказuid=33(www-data) Модульrce CVSSCVSS 9.8 — Critical

🔎 Що це таке

Виконання довільного коду на сервері. Найнебезпечніша категорія вразливостей. Зловмисник отримує повний контроль над сервером: доступ до файлів, БД, мережевих ресурсів, може встановити backdoor або ransomware.

💥 Можливий вплив

Повний компроміс сервера, витік усіх даних, потенційне поширення в мережі.

🛡 Як виправити

Негайно ізолюйте сервер. Оновіть фреймворк/залежності. Ніколи не передавайте дані від користувача у eval(), exec(), system(). Використовуйте allowlist для дозволених операцій.

SQL Injection https://demo.site.com/user → id Критична ▶
URLhttps://demo.site.com/user Параметрid Payload' ДоказSQL syntax error near ''' Модульsqli CVSSCVSS 9.8 — Critical

🔎 Що це таке

SQL-ін'єкція дозволяє маніпулювати запитами до бази даних. Зловмисник може читати, змінювати або видаляти будь-які дані в БД, обходити автентифікацію, а в деяких конфігураціях — виконувати команди ОС.

💥 Можливий вплив

Витік усіх даних БД (паролі, персональні дані, фінансова інформація), обхід авторизації, пошкодження даних, повний компроміс сервера (через xp_cmdshell/UDF).

🛡 Як виправити

Використовуйте параметризовані запити (prepared statements). Ніколи не конкатенуйте SQL з даними користувача. Застосовуйте ORM. Обмежте права облікового запису БД.

Local File Inclusion https://demo.site.com/file → path Критична ▶
URLhttps://demo.site.com/file Параметрpath Payload../../../../etc/passwd Доказroot:x:0:0:root:/root:/bin/bash Модульlfi CVSSCVSS 8.6 — High

🔎 Що це таке

Вразливість дозволяє читати довільні файли сервера через path traversal. Зловмисник отримує доступ до /etc/passwd, конфігураційних файлів, приватних ключів SSH, вихідного коду застосунку та логів.

💥 Можливий вплив

Витік облікових даних, приватних ключів, вихідного коду. При певних умовах може призвести до RCE (через log poisoning).

🛡 Як виправити

Ніколи не використовуйте дані користувача у шляхах файлів. Застосовуйте allowlist дозволених файлів. Обмежте права читання файлів для веб-процесу. Використовуйте chroot/container ізоляцію.

Server-Side Request Forgery (SSRF) https://demo.site.com/fetch → url Висока ▶
URLhttps://demo.site.com/fetch Параметрurl Payloadhttp://169.254.169.254/ ДоказHTTP/1.1 200 OK (AWS metadata) Модульssrf CVSSCVSS 8.6 — High

🔎 Що це таке

Сервер виконує HTTP-запити на довільні адреси за вказівкою зловмисника. Це дозволяє сканувати внутрішню мережу, звертатися до метадата-сервісів хмарних провайдерів (AWS/GCP/Azure), обходити файрволи та отримувати токени доступу до хмарної інфраструктури.

💥 Можливий вплив

Доступ до внутрішніх сервісів, хмарних метаданих (IAM-ролі, токени AWS), сканування внутрішньої мережі, обхід захисту периметра.

🛡 Як виправити

Не дозволяйте серверу робити запити за URL від користувача без валідації. Використовуйте allowlist дозволених доменів. Відключіть непотрібні схеми (file://, gopher://, dict://). Ізолюйте сервер від внутрішньої мережі через egress firewall.

Cross-Site Scripting (XSS) https://demo.site.com/search → q Висока ▶
URLhttps://demo.site.com/search Параметрq Payload<script>alert(1)</script> ДоказPayload reflected unescaped in body Модульxss CVSSCVSS 7.4 — High

🔎 Що це таке

Cross-Site Scripting дозволяє впровадити шкідливий JavaScript у сторінку, яку бачать інші користувачі. Зловмисник може вкрасти cookies/сесії, виконувати дії від імені жертви, перенаправляти на фішинг-сторінки, перехоплювати введення форм (keylogging).

💥 Можливий вплив

Крадіжка сесій та облікових даних, defacement, поширення шкідливого ПЗ, виконання привілейованих дій від імені адміністратора.

🛡 Як виправити

Екранізуйте весь вивід у HTML (< → <, > → > тощо). Використовуйте Content Security Policy (CSP). Встановіть прапори HttpOnly та Secure для cookies. Застосовуйте бібліотеки для безпечного рендерингу (DOMPurify).
